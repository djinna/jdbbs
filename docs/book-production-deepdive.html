Technical Analysis

Prodcal is a production coordinator, a manuscript intake system, and a thin book-build orchestrator.

The repo combines project scheduling, client/project auth, manuscript transmittal capture, book spec normalization, and two output paths: a Typst-based interior PDF compiler and a Pandoc-based EPUB generator.

Primary stack
Go + SQLite + static HTML/JS
Production assets
DOCX in, PDF/EPUB blobs out
Core bridge
Transmittal -> book spec -> build config
Code paths reviewed: srv/transmittal.go, srv/books.go, srv/epub.go, srv/bookspecs.go, srv/server.go, srv/static/admin.html, and migrations under db/migrations/.
Executive Read

The product model is coherent: editorial intake drives structured production metadata, which drives both schedule and output generation.

What the system does well

It treats the manuscript transmittal as first-class operational data instead of an email attachment or static PDF.
It binds project schedule, asset flow, editorial notes, and build settings to the same project record.
It creates a real metadata bridge from intake fields into downstream typesetting and EPUB settings.
It uses static frontends with a small Go server, which keeps deployment and maintenance surface area low.

Where the implementation is thin

Build execution depends on an external filesystem subtree at /home/exedev/book-production, so the repo is not self-contained.
Conversion jobs run as in-process goroutines with no durable queue, retries, ownership, or audit trail.
Output storage is blob-in-SQLite, which is workable for small scale but weak for large catalogs and large interiors.
Security is inconsistent: the book download endpoint is not protected even though book listing and conversion are admin-only.
End-to-End Flow

From manuscript transmittal to deliverables

This is the actual flow encoded in the repo, not an abstract publishing lifecycle.

1. project shell
Create a project with client/project slugs, dates, auth, tasks, and a client portal route.
2. transmittal intake
Auto-create a structured JSON transmittal, then save through the SPA with version snapshots.
3. spec normalization
Pull selected transmittal fields into a project-level book spec used by print and EPUB paths.
4. source ingest
Upload DOCX into the books table, optionally link it to the project, then compile outputs.
5. distribution
Download generated PDF/EPUB, track files, log journal items, and manage corrections/export.

Intake Layer

The transmittal subsystem is the strongest product idea in the repo.

Why it matters

1

Structured intake

The default JSON schema captures the real publishing handoff: front matter, back matter, art, permissions, design, files, and proofs.
2

Continuous editing

transmittal.js autosaves with a 600ms debounce, so the form behaves like a working document rather than a submit-once artifact.
3

Recoverability

Older states are stored in transmittal_versions, which is enough for practical restore operations without needing a complex revision engine.
4

Cross-project reuse

The duplicate endpoint lets operators roll a transmittal forward into another project, which fits series or edition workflows.

Print Path

The PDF pipeline is a stitched toolchain: Pandoc -> Python transform -> Typst template -> compiled interior PDF.

Actual execution sequence

1

DOCX stored as blob

handleUploadBook writes source bytes into books.source_data.
2

Pandoc conversion

pandoc --from=docx+styles --to=markdown+fenced_divs creates intermediate markdown.
3

Python transform

md-to-chapter.py converts markdown into a Typst chapter file.
4

Main Typst assembly

runConversion writes a main.typ that imports the external series template and injects project overrides.
5

Typst compile

typst compile --font-path ... renders output.pdf, which is then stored in books.pdf_data.
Print pipeline dependencies
bookProdRoot = /home/exedev/book-production
convertScript = /home/exedev/book-production/scripts/md-to-chapter.py

optional symlinks:
  styles.typ

Important inference: this repo orchestrates print compilation, but the real design/template system lives outside the repo. That is the biggest architectural dependency in the whole product.
Digital Path

The EPUB path is simpler and cleaner than print: DOCX -> Pandoc EPUB3 with metadata, CSS, and optional cover.

EPUB generation behavior

Starts from the same stored DOCX source.
Loads project-linked book spec when present.
Writes cover blob to a temp file if one exists.
Builds CSS from spec settings plus freeform user CSS.
Calls pandoc --to=epub3 --toc with metadata, cover, and stylesheet options.
Stores the resulting binary in books.epub_data.

Compared with the print path, this is less bespoke and more portable. It is also less expressive because it relies on Pandoc EPUB generation rather than a richer digital composition layer.
Operational Layer

Prodcal is not only a compiler trigger. It also captures the handoff, propagation, and communication work around production.

Schedule + budget

Projects and tasks remain the top-level organizing model.

File log

file_log records inbound/outbound transfers, file types, recipients, and notes.

Journal

journal stores calls, decisions, approvals, and notes for project memory.

Corrections

corrections tracks discrete find/replace changes and exports them as YAML.

Why this matters

The repo understands that book production is not just rendering. It is coordination, approvals, and asset propagation.
The file log and journal make the product more useful to small production teams than a build-only tool would be.
The corrections export suggests a path toward structured downstream amendment workflows.

Current limit

These operational records are adjacent to the build pipeline, but not deeply wired into it.
Corrections are exported, but not automatically applied or reconciled against source/build versions.
There is no explicit artifact lineage linking a generated PDF/EPUB to a transmittal version, correction set, or exact spec revision.
Technical Risks

The biggest gaps are durability, security, and provenance.

Unauthenticated book download endpoint

handleDownloadBook does not call requireExeDevAdminAPI or requireAuth.
Impact: if IDs are guessable, uploaded/generated book files are publicly retrievable.

In-process background jobs only

Both conversion paths launch goroutines directly from request handlers.
Impact: no persistence across restarts, no queueing, no backpressure, no recovery semantics.

External build system dependency

Templates, fonts, and scripts live outside the repo at a fixed absolute path.
Impact: reproducibility is weak; local dev and disaster recovery depend on hidden state.

No typed schemas for JSON payloads

Transmittal and spec payloads are manipulated through map[string]any and raw JSON strings.
Impact: silent drift, weak validation, brittle migrations.

Blob-heavy SQLite design

DOCX, PDF, EPUB, and covers all live in SQLite rows.
Impact: acceptable at low scale, but not ideal for large assets, retention, or artifact serving.

Weak artifact provenance

Outputs are not explicitly tied to spec version, transmittal version, correction export, or toolchain version.
Impact: hard to answer "what exactly produced this file?"
Recommended Next Architecture

If the goal is a stronger production platform, keep the product model and harden the execution model.

1. Secure and version artifacts

Protect all book downloads.
Create an artifacts table or object store layer.
Stamp artifacts with source book ID, spec hash, transmittal version, and toolchain version.

2. Introduce durable jobs

Replace request-spawned goroutines with a persisted job queue.
Track queued/running/failed/completed states and stderr.
Add retry semantics and operator-visible run history.

3. Formalize schemas

Define typed Go structs for transmittal and book spec.
Validate payloads server-side.
Make transmittal -> spec mapping explicit, testable, and migratable.

What I would preserve

The transmittal-first product concept
The project-level coordination model
The clean static frontend approach
The spec bridge between editorial and rendering concerns

What I would change first

Lock down downloads immediately
Move build dependencies inside a reproducible package or container image
Persist build jobs and artifact lineage
Add integration tests around transmittal mapping and both output pipelines
Bottom line: the repo already contains a credible production-system thesis. The work now is less about inventing features and more about making the execution layer trustworthy.
